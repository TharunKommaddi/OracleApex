# Procedure: PCK_RI_HISTORY.psetEndDateVorgaenger

This procedure is part of the `PCK_RI_HISTORY` package and is responsible for setting the appropriate "expiration" or "out of force" dates for predecessor regulations when new implementation dates (Einsatzdaten) are entered for a successor regulation.

## Parameters

- `avorschriftid` (NUMBER): The unique identifier of the current (successor) regulation.
- `anotvorschriftid` (NUMBER, optional): The unique identifier of a predecessor regulation to be excluded from the calculation, if needed.

## Tables Involved

- `T_VORSCHRIFT_REF_VORSCHRIFT`: This table stores the relationships between predecessor and successor regulations.
- `T_VORSCHRIFT_REF_EINSATZDATUM_TYP`: This table contains the implementation dates (Einsatzdaten) for various regulations and implementation date types (EINSATZDATUM_TYPID).

## Process Flow

1. Retrieve all predecessor regulations (`vorgaenger_vorschriftid`) for the current regulation (`avorschriftid`) from the `T_VORSCHRIFT_REF_VORSCHRIFT` table, where the relationship type (`beziehung_art`) is 10 (indicating a predecessor-successor relationship), and excluding the regulation ID passed in `anotvorschriftid` (if provided).

2. For each predecessor regulation, fetch the implementation dates (`einsatzdatum`) for the successor regulation (`avorschriftid`) from the `T_VORSCHRIFT_REF_EINSATZDATUM_TYP` table, separating the dates for "All Vehicles" (`einsatzdatum`) and "New Types" (`neue_typen`).

3. Calculate the minimum implementation dates for "All Vehicles" (`einsatzdatum`) and "New Types" (`neue_typen`) for each predecessor regulation.

4. Determine the `Datum_type` based on the `einsatzdatum_typid` values:
   - 1023 for "All Vehicles FBU" = 1060
   - 1021 for "All Vehicles CKD" = 1063
   - Everything else is 1041

5. Calculate the final `einsatzdatum` (implementation date - 1 day) and `hinweis` (warning message) for each predecessor regulation.

6. Join with the `T_VORSCHRIFT_REF_EINSATZDATUM_TYP` table to retrieve the `MANUELLE_EINTRAGUNG` value (if it exists) and filter for records where `MANUELLE_EINTRAGUNG` is not set or is equal to 20.

7. Update or insert a row in the `T_VORSCHRIFT_REF_EINSATZDATUM_TYP` table for each predecessor regulation with the calculated `einsatzdatum`, `hinweis`, and `EINSATZDATUM_TEXT` set to 'System Berechnung über Nachfolger'.

```
2384         | 1021               | 01.07.2024
```

Now, let's call the procedure:

```sql
BEGIN
  PCK_RI_HISTORY.psetEndDateVorgaenger(2384, 2385);
END;

```

Here, we're passing `2384` as the `avorschriftid` (the current regulation ID) and `2385` as the `anotvorschriftid` (the regulation ID to exclude, if needed).

After running the procedure, we should check the `T_VORSCHRIFT_REF_EINSATZDATUM_TYP` table for the predecessor regulation (2383) to see if the implementation dates have been updated correctly.

# Procedure: PCK_RI_HISTORY.pSetEndDateVorschrift

This procedure is also part of the `PCK_RI_HISTORY` package and is responsible for calling the `psetEndDateVorgaenger` procedure for the current regulation and any successor regulations that might be affected by the new implementation dates.

## Parameters

- `avorschriftid` (NUMBER): The unique identifier of the current regulation.

## Process Flow

1. Retrieve the current regulation ID (`avorschriftid`) and any successor regulation IDs (`nachfolger_vorschriftid`) from the `T_VORSCHRIFT_REF_VORSCHRIFT` table, where the relationship type (`beziehung_art`) is 10 (indicating a predecessor-successor relationship) and the `VORGAENGER_VORSCHRIFTID` matches the current regulation ID (`avorschriftid`).

2. For each regulation ID retrieved (the current regulation ID and any successor regulation IDs), call the `psetEndDateVorgaenger` procedure, passing the regulation ID as `avorschriftid` and the current regulation ID (`avorschriftid`) as `anotvorschriftid`.

This procedure ensures that the "expiration" or "out of force" dates are calculated and updated not only for the predecessor regulations of the current regulation but also for any successor regulations that might be affected by the new implementation dates.

By calling both procedures in the appropriate order, the system ensures that the implementation date changes are propagated correctly throughout the predecessor-successor relationships, updating the relevant dates and warning messages in the `T_VORSCHRIFT_REF_EINSATZDATUM_TYP` table.
