This is the start of the `pSetEndDateVorschrift` procedure. It takes one parameter: `avorschriftid` (the current regulation ID).

```sql
for c in (
  Select avorschriftid as vorschriftid from dual
  union
  Select nachfolger_vorschriftid from t_vorschrift_ref_vorschrift tvrv
  where tvrv.beziehung_art = 10 and VORGAENGER_VORSCHRIFTID = avorschriftid
)
loop
  psetEndDateVorgaenger(c.vorschriftid, anotvorschriftid => avorschriftid);
end loop;
```

This part of the code retrieves a set of regulation IDs, including the current regulation ID (`avorschriftid`) and any successor regulation IDs (`nachfolger_vorschriftid`) from the `t_vorschrift_ref_vorschrift` table, where the relationship type (`beziehung_art`) is 10 (indicating a predecessor-successor relationship) and the `VORGAENGER_VORSCHRIFTID` matches the current regulation ID (`avorschriftid`).

It then loops through each regulation ID in this set and calls the `psetEndDateVorgaenger` procedure, passing the regulation ID as `avorschriftid` and the current regulation ID (`avorschriftid`) as `anotvorschriftid`.

This ensures that the "expiration" or "out of force" dates are calculated and updated not only for the predecessor regulations of the current regulation but also for any successor regulations that might be affected by the new implementation dates.

```sql
end pSetEndDateVorschrift;
```
